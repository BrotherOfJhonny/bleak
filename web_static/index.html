❓ Ajuda & Documentação — BLEAK V0.19

🧭 O que é o BLEAK?

BLEAK — Bluetooth Link Exploitation & Attack Knowledgebase é uma plataforma integrada de segurança Bluetooth projetada para Red Team, Pentest e pesquisa ofensiva.

Combina descoberta passiva, enumeração GATT, varredura de vulnerabilidades, exploits ativos, vigilância comportamental (BlueHood), e assessment de Fast Pair (CVE-2025-36911 WhisperPair) em uma única interface web. Usa Realtek RTL8761BUV via raw HCI para spam (todas as plataformas). ESP32-S3 para HID injection e fallback NimBLE.

🧩 Compatibilidade de Hardware

O BLEAK detecta adaptadores e capacidades em tempo de execução. Ele pode rodar em setups diferentes, mas cada módulo depende de recursos específicos.

Perfil mínimo:
• Kali/Debian/Ubuntu com BlueZ ativo
• Python 3.12+
• Um adaptador Bluetooth exposto como hci0, hci1, etc.
• Ferramentas básicas: bluetoothctl, hciconfig, btmgmt
Com esse perfil, normalmente funcionam Discovery, GATT Explorer, Vulnerability Scan básico, PoC BLE, Fast Pair scan via BlueZ e relatórios.

Perfil completo recomendado:
• Adaptador HCI USB confiável para BlueZ
• ESP32-C3 ou ESP32-S3 com firmware BLEAK
• PipeWire ou PulseAudio funcional no usuário gráfico
• Ferramentas de áudio: wpctl, pactl, pw-record, parecord
• Ferramentas opcionais: bettercap, sdptool, hcitool, bluez-tools

Dependências por módulo:
• Discovery, GATT, Vuln Scan: HCI/BlueZ basta.
• Fast Pair Scanner: HCI funciona; ESP32 melhora detecção de anúncios que o BlueZ pode filtrar.
• WhisperPair verificação: HCI/BlueZ; ESP32 opcional para scan.
• WhisperPair gravação: HCI/BlueZ + PipeWire/PulseAudio + endpoint Classic/A2DP ou HFP visível.
• BlueSpy: btmgmt, bluetoothctl, pactl/parecord.
• BLE Spam, Karma, Beacon, MAC Clone: dependem da ESP32 para o fluxo completo.
• HID Injection: requer ESP32-S3 ou hardware compatível.
• Relatórios: independem de hardware e incluem evidências de áudio arquivadas.

Limitações comuns em outras máquinas:
• A ESP32 pode aparecer em outra porta (/dev/ttyUSB0, /dev/ttyACM0, etc.).
• Dongles diferentes variam no suporte BLE, Classic BT e A2DP.
• A vulnerabilidade pode ser comprovada, mas a gravação falhar se PipeWire/PulseAudio não criar bluez_card.* ou um nó wpctl Bluetooth.
• Fones com MAC rotativo podem aparecer com múltiplos endereços; evidências de áudio confirmadas são arquivadas para relatório.
• Se o firmware ESP32 não for o BLEAK esperado, os módulos ESP32 podem não responder.

📋 Pipeline de Assessment BLE

  1. 📡 Discovery — Scan passivo BLE/Classic via adaptador HCI ativo com radar em tempo real. Identifica device, RSSI, domínio (IoT, wearable, smartphone, audio). Timeout configurável.
  2. 🔌 Enumeração GATT — Conecta e lê todos os serviços e características. Gera exposure score 0-100. UUIDs resolvidos (incluindo 0xFE2C = Fast Pair Service).
  3. 🛡 Vuln Scan — 56 checks passivos: Just Works, Unencrypted Read, Writable Without Auth, DFU Exposed, Static MAC, Ad Leakage, e outros. Perfis: Pentest Básico, Smartwatch e IoT.
  4. ⚡ Exploit/PoC — Exploits ativos confirmam vulnerabilidades. Gera scripts Python. Job-based com polling em tempo real. Inclui CVE-2023-45866 (BlueDucky), BLE-SPAM, RFCOMM probe.
  5. 🎧 Audio / WhisperPair — Fast Pair Scanner (0xFE2C / 0x00E0), WhisperPair Flow, KBP Multi-Strategy V14, Find Hub Check. Assessment completo de CVE-2025-36911.
  6. 👁 BlueHood Surveillance — Vigilância passiva contínua. Classificação automática, heatmap horário, correlação comportamental, alvos Fast Pair em tempo real.
  7. 📋 Relatório — Técnico (detalhado com evidências) ou Gerencial (Risk Score, Business Impact, Plano de Ação). Formatos: HTML, JSON, CSV.

🎧 Audio Security — WhisperPair CVE-2025-36911

Fast Pair Scanner: Detecta devices com UUID 0xFE2C e manufacturer data Google (0x00E0). Prioriza ESP32-C3 (detecta devices pareados que BlueZ filtra). Fallback: Bleak discover(return_adv=True) + bluetoothctl cache.

WhisperPair Flow: Scan → Detecção multi-estratégia (Discovery cache → BlueZ info → ESP32-C3 → Bleak scan → deauth rescan → direct connect) → GATT connect → Service discovery → KBP write → verificar resposta → VULNERABLE / PATCHED.

KBP Multi-Strategy V14: Testa estratégias em cascata, incluindo o payload compacto do zwhisper:
  ① RAW_KBP_16B — 16 bytes simples, flags=0x00 (implementações mais quebradas)
  ② ZWHISPER_65B — type 0x00 + 64B public key dummy
  ③ RAW_WITH_PUBKEY — 16B header + 64B ECDH dummy = 80 bytes
  ④ FLAGS_INITIATE — flags=0x11 (INITIATE_BONDING | EXTENDED_RESPONSE)
Response parsing extrai: msg_type, BR/EDR encrypted address, salt.

Find Hub Check: Detecta se Account Key char (0xFE2C1238/1235) é writable sem autenticação → risco de o atacante se tornar owner via Google Find Hub (location tracking).

Dispositivos vulneráveis confirmados: Sony, Google, JBL, Jabra, Anker/Soundcore, Nothing, OnePlus, Xiaomi, Marshall, Logitech, Edifier.

👁 BlueHood Surveillance

Inspirado em BlueHood (dannymcc), o módulo de vigilância passiva escaneia continuamente sem nunca conectar a nenhum device.

Classificação automática: OUI lookup (70+ vendors), UUID mapping (12 serviços BLE), análise de nome. Categorias: Phone, Audio, Wearable, Vehicle, IoT, HID, Fast Pair.

Correlação comportamental: Devices vistos juntos ≥3 ciclos → provavelmente mesmo usuário (ex: phone + earbuds). Revela padrões de presença sem conectar em nenhum device.

Heatmap horário: Rastreia em que horas cada device aparece. Sessões detectadas por gap > 5 min.

MACs aleatórios: Filtrados automaticamente (locally administered bit) — reduz ruído de phones modernos.

Alvos Fast Pair: Devices 0xFE2C listados separadamente com acesso direto a WP Flow, KBP V14, Find Hub.

🎙 BlueSpy — BSAM-PA-05 (Eavesdropping)

Vulnerabilidade: Device aceita pairing Bluetooth sem confirmação do usuário (Just Works / NoInputOutput IO Capability). Um atacante próximo pode se parear, acessar perfis HFP/A2DP e gravar o microfone sem nenhuma notificação visual.

Controle violado: BSAM-PA-05 — "Pairing sem interação do usuário" (Bluetooth Security Assessment Methodology, Tarlogic)

Fluxo de 2 fases no BLEAK:
① Fase 1 — Verificação: Define IO Capability para NoInputOutput via btmgmt io-cap 3 → tenta btmgmt pair -c 3 -t 0 MAC → se aceito: BSAM-PA-05 violado
② Fase 2 — Evidência: Conecta perfil A2DP/HFP → detecta fonte de áudio via pactl list sources → grava via parecord → evidência disponível para download e relatório

Ferramentas necessárias: btmgmt, bluetoothctl, pactl, parecord (pacote bluez-utils + libpulse)

⚠ PulseAudio/PipeWire e sudo:
O servidor BLEAK roda com sudo (necessário para btmgmt e hciconfig), mas parecord precisa do socket do usuário (/run/user/<uid>/pulse) que não existe para root. O BLEAK resolve isso automaticamente detectando o usuário original via $SUDO_USER e executando parecord com preexec_fn=setuid/setgid no contexto correto. Se falhar, use o comando manual fornecido sem sudo em outro terminal.

Interpretação dos resultados:
🔴 VULNERABLE (pair + áudio) — Eavesdropping totalmente confirmado com evidência de áudio
🟡 VULNERABLE (pair sem áudio) — BSAM-PA-05 violado; áudio requer conexão A2DP manual
🟢 PATCHED — Device exige confirmação do usuário para parear

Referência: TarlogicSecurity/BlueSpy | BSAM-PA-05

🔬 RACE / Airoha — CVE-2025-20700/20701/20702

Vulnerabilidade: Chips Airoha Technology Corp. (série AB1562/AB1563) expostos em dezenas de marcas de headphones e earbuds expõem o protocolo proprietário RACE via BLE GATT e BT Classic RFCOMM sem autenticação.

CVEs:
  • CVE-2025-20700 — RACE via BLE GATT sem auth → RAM read, flash dump, link key extraction
  • CVE-2025-20701 — RACE via BT Classic RFCOMM sem auth → pivoting do BLE para Classic
  • CVE-2025-20702 — Exposição geral do protocolo RACE em ambos transportes

CVSS Base Score: 8.8 (HIGH) — CVE-2025-20700 | Vetor: AV:A/AC:L/PR:N/UI:N/S:U/C:H/I:H/A:H

Marcas afetadas (chips Airoha): Sony WH-1000XM5, WH-1000XM4, WF-1000XM5, JVC, Pioneer, Anker Soundcore, Jabra Evolve, Technics (Panasonic), Audio-Technica ATH-, Denon, 1More, FiiO, TOZO, Edifier

Fluxo de detecção no BLEAK:
① Fingerprint passivo — OUI prefixo Airoha + nome do device
② GATT connect → busca UUIDs 0x0000AE00–AE07 (serviço RACE proprietário)
③ Envia RACE_CMD_GET_SDK_INFO sem autenticar → response = CVE-2025-20700 confirmado
④ sdptool browse → procura RACE RFCOMM service → CVE-2025-20701

Comandos RACE disponíveis após confirmação:
  • sdkinfo — versão do SDK Airoha no chip
  • buildversion — build string do firmware (fingerprint de versão)
  • bdaddr — obtém Classic BT address sem precisar de sniffer (ex: Ubertooth)
  • flash_header — dump de 256 bytes da flash (extração de link keys via NVDM)

Referência: auracast-research/race-toolkit | Auracast Research Blog

📡 Discovery & GATT Explorer

Discovery: BleakScanner via BlueZ/HCI ativo. Radar BLE em tempo real com posição por distância estimada (RSSI→m). Cores por domínio. Scan Classic BT via hcitool e bettercap.

GATT Explorer: Job-based (non-blocking) — retorna job_id imediatamente, frontend faz polling. Sem timeout de browser. UUID 0xFE2C resolvido como "Fast Pair Service". Características Fast Pair (FE2C1233–1236) identificadas por nome.

Erros comuns:
• "Device not found" → Execute Discovery primeiro, ou use bluetoothctl scan on para registrar no BlueZ.
• "No discovery started" → Pare o Discovery antes de usar o Fast Pair Scanner.
• "Service Discovery not performed" → Retry. Device tem timeout agressivo.

🔑 ESP32-C3 — Fast Pair & Novos Comandos AT

O ESP32-C3 é o adapter principal para Fast Pair scanning — captura advertisements de devices já pareados que BlueZ filtra.

Novos comandos AT (firmware v4.2+):
AT+FPSCAN=<sec> — Scan dedicado Fast Pair. Filtra 0xFE2C e 0x00E0 em hardware. Emite: FP:MAC:RSSI:modelID:state:name
AT+SCANRAW=<sec> — Scan raw com payload hex completo. Para análise de advertisement custom.
AT+SCAN=<sec> — Scan geral. Emite: DEV:MAC:RSSI:name
AT+VERSION — Versão do firmware
AT+STATUS — Estado atual
AT+HELP — Lista todos os comandos

Fix de compilação: NimBLE ≥1.4: uuid.toString().substr(4,4) em vez de getNative().

⚠️ Troubleshooting

Adaptador HCI indisponível: sudo hciconfig up ou sudo rfkill unblock bluetooth
Device não encontrado no WhisperPair: Execute Discovery → clique no device → WhisperPair Flow. O sistema tenta 6 estratégias de detecção incluindo deauth + rescan.
Device conectado ao celular: Desconecte manualmente no celular (não despareie) → aguarde LED piscar → tente novamente.
Fast Pair Scanner timeout: É job-based — o timer na tela indica progresso real. Se ESP32-C3 não conectado, usa BlueZ (pode não ver devices pareados).
NetworkError nos botões: Verifique se o servidor está rodando (sudo ./run_web_lan.sh) e se não há erro Python no terminal.
Ferramentas externas: sudo ./install.sh | Status: ./status.sh
